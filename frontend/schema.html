Schema, decisions, trade-offs.

13 tables in 3NF. Every split, every constraint, every index here is justified by either a normalization rule or a measured query path - no decoration. Read this page as "why this shape and not another".

01

Why 13 tables and not 4-5 fat ones

Each split below is justified by a specific normalization rule. Collapsing any of them would either violate 1NF (repeating groups), 2NF (partial dependency on a composite key) or 3NF (transitive dependency through a non-key attribute). The price we pay is JOINs at read time; the price we'd otherwise pay is update anomalies and storage redundancy.

Layer 1

Independent entities

No outgoing FKs. The "things in the world" the system models.

  • UserPK: user_id, candidate: email
  • Event_CategoryPK: category_id, candidate: category_name
  • LocationPK: location_id
  • HostPK: host_id, candidate: contact_email
  • PerformerPK: performer_id
  • OfferPK: offer_id, candidate: code
Layer 2

Dependent entities

Hold the business core. FKs into Layer 1.

  • EventFK to Category, Location, Host
  • BookingFK to User, Event
  • TransactionFK to Booking (UNIQUE = 1:1)
  • ReviewFK to User, Event; UNIQUE(user_id,event_id)
Layer 3

Bridge tables (M:N)

Composite PK on the two FK columns. 2NF/3NF by construction.

  • Saved_ItemUser × Event
  • Event_PerformerEvent × Performer
  • Booking_OfferBooking × Offer
02

Normalization, with worked examples

Three concrete cases. Each shows the de-normalized shape, what would have gone wrong, and the fix.

1NF

Performer on Event — the repeating-group trap

Naive shape
Event(event_id, title, ..., performer_csv VARCHAR(500))
-- or worse:
Event(event_id, title, ..., performer_1, performer_2, performer_3)

CSV violates 1NF (atomicity). The numbered columns also violate 1NF (repeating group) and force an arbitrary cap on lineup size.

3NF fix
Performer(performer_id, name, performer_type)

Event_Performer(event_id, performer_id, performance_order)
PRIMARY KEY (event_id, performer_id)

Bridge table. Unbounded lineup, queryable by performer (find every event a performer has done), one canonical performer record.

3NF

Venue address on Event — the transitive-dependency trap

Naive shape
Event(event_id, title, venue_name, city,
      state, locality, pin_code,
      max_event_capacity, ...)

venue_name → (city, state, pin_code, capacity). Two events at the same venue duplicate every address attribute. Fixing a typo means an UPDATE across N events.

3NF fix
Location(location_id, venue_name, city,
         state, locality, pin_code,
         max_event_capacity)

Event(event_id, title, ..., location_id INT)
FOREIGN KEY (location_id) REFERENCES Location(location_id)

Address attributes depend only on location_id. One venue, one row. Pay one JOIN, gain one source of truth.

3NF

Offer details on Booking — the same trap, again

Naive shape
Booking(booking_id, ..., offer_code,
        offer_type, offer_value)

offer_code → (offer_type, offer_value). Editing a campaign means rewriting every booking that used it — corrupts historical receipts.

3NF fix
Offer(offer_id, code, type, discount_value, ...)

Booking_Offer(booking_id, offer_id,
              discount_applied DECIMAL)
PRIMARY KEY (booking_id, offer_id)

Offer terms in one table. discount_applied is the rupees actually deducted at redemption time — deliberately frozen, so historical bookings stay correct even if the offer changes.

03

Honest denormalizations

Two places we deliberately did not normalize. Calling them out so they're not mistaken for oversights.

User.city & User.occupation

Could be FKs to City / Occupation lookup tables. We left them as VARCHAR. The lookup tables would have one column each (just the name) and give us nothing back except an extra JOIN. Strict 3NF still holds because both attributes depend only on user_id.

Booking.total_price

Computable from Event.price × number_of_people - discount, so arguably a transitive dependency. Stored anyway because it locks in the price at booking time — if the event price changes later, the booking still reflects what the user paid. Refunds and audits need the as-charged amount, not the current one. Documented denormalization for auditability.

04

Keys at a glance

KindDefinitionWhere in our schema
Primary key Uniquely identifies a row, NOT NULL. Every table - <table>_id INT AUTO_INCREMENT for entities, composite (FK1, FK2) for bridges.
Foreign key References a PK in another table; enforces referential integrity. Event → Category/Location/Host; Booking → User/Event; Review → User/Event; all bridges.
Candidate key Could have been the PK. Enforced with UNIQUE. User.email, Event_Category.category_name, Host.contact_email, Offer.code.
Composite key PK formed by two or more columns. All three bridge tables: (user_id, event_id), (event_id, performer_id), (booking_id, offer_id).
Composite UNIQUE Business rule expressed at the schema level. Review UNIQUE (user_id, event_id) — one review per user per event.
